Windows 命名管道（named pipes）是一种用于管道服务器与一个或多个管道客户端之间通信的具名单向或双向管道。

命名管道可用于在同一台计算机上的进程之间或在网络中不同计算机上的进程之间提供通信。

用于实例化命名管道的服务器端函数 `CreateNamedPipe()`。 用于接受连接的服务器端函数 `ConnectNamedPipe()`。 客户端进程使用 `CreateFile()` 或 `CallNamedPipe()` 函数连接到命名管道。

## 命名管道名

每个命名管道都有一个唯一的名称，用于将其与系统命名对象列表中的其他命名管道区分开来。 管道服务器在调用 `CreateNamedPipe()` 函数以创建命名管道的一个或多个实例时指定管道的名称。 管道客户端在调用 `CreateFile()` 或 `CallNamedPipe()` 函数以连接到命名管道的实例时指定管道名称。

在 `CreateFile()`、`WaitNamedPipe()` 或 `CallNamedPipe()` 函数中指定管道的名称时，使用以下格式：

```
\\ServerName\pipe\PipeName
```

`ServerName` 是远程计算机的名称或用于指定本地计算机的句点。 `PipeName` 指定的管道名称字符串可以包含除反斜杠以外的任何字符，包括数字和特殊字符。 整个管道名称字符串最多可以包含 256 个字符。 管道名称不区分大小写。

管道服务器不能在另一台计算机上创建管道，因此 `CreateNamedPipe()` 必须使用服务器名称的句点，如以下示例：

```
\\.\pipe\PipeName
```

管道服务器可以向其管道客户端提供管道名称，以便它们可以连接到管道。 管道客户端可以从某些持久化的来源（例如注册表项、文件或其他应用程序）获取管道名称。 否则，客户端必须在编译时知道管道名称。

## 命名管道打开模式

管道服务器在 `CreateNamedPipe()` 函数的 `dwOpenMode` 参数中指定管道访问、重叠和写通模式。 管道客户端可以使用 `CreateFile()` 函数为管道句柄指定这些打开模式。

### 访问模式（Access Mode）

设置管道访问模式相当于指定与管道服务器句柄相关的读取或写入访问权限。 下表显示了可以使用 `CreateNamedPipe()` 指定的每个访问模式的等效通用访问权限。

| 访问模式               | 等效的通用访问权限                |
| :--------------------- | :-------------------------------- |
| `PIPE_ACCESS_INBOUND`  | `GENERIC_READ`                    |
| `PIPE_ACCESS_OUTBOUND` | `GENERIC_WRITE`                   |
| `PIPE_ACCESS_DUPLEX`   | `GENERIC_READ` \| `GENERIC_WRITE` |

- `PIPE_ACCESS_INBOUND`：该管道对于管道服务器来说是只读的，对于管道客户端来说是只写的。
- `PIPE_ACCESS_OUTBOUND`：该管道对于管道服务器来说是只写的，对于管道客户端来说是只读的。
- `PIPE_ACCESS_DUPLEX`：对于管道服务器和管道客户端来说都是可读可写的。

使用 `CreateFile()` 连接到命名管道的客户端必须在 `dwDesiredAccess` 参数中指定与管道服务器指定的访问模式兼容的访问权限。例如，客户端必须指定 `GENERIC_READ` 访问权限才能打开管道服务器使用 `PIPE_ACCESS_OUTBOUND` 创建的管道的句柄。对于管道的所有实例，访问模式必须相同。

若要读取管道属性（如读取模式或阻塞模式），管道句柄必须具有 `FILE_READ_ATTRIBUTES` 访问权限；若要写入管道属性，管道句柄必须具有 `FILE_WRITE_ATTRIBUTES` 访问权限。这些访问权限可以与适用于管道的通用访问权限组合：对于只读管道，使用 `GENERIC_READ` 与 `FILE_WRITE_ATTRIBUTES`；对于只写管道，使用 `GENERIC_WRITE` 与 `FILE_READ_ATTRIBUTES`。以这种方式限制访问权限可为管道提供更好的安全性。

### 安全性和访问权限

在调用 `CreateNamedPipe()` 函数时，可以为命名管道指定一个安全描述符。该安全描述符控制对命名管道客户端端口和服务器端口的访问。如果您指定 `NULL`，则命名管道将获得默认的安全描述符。命名管道默认安全描述符中的访问控制列表授予 `LocalSystem` 账户、管理员和创建者所有者完全控制权。它们还授予 `Everyone` 组成员和匿名账户读取访问权限。

若要检索命名管道的安全描述符，请调用 `GetSecurityInfo()` 函数。若要更改命名管道的安全描述符，请调用 `SetSecurityInfo()` 函数。

当线程调用 `CreateNamedPipe()` 打开现有命名管道服务器端的句柄时，系统会在返回句柄之前执行访问检查。该访问检查会将线程的访问令牌和请求的访问权限与命名管道的安全描述符中的 DACL 进行比较。除了请求的访问权限外，DACL 还必须允许调用线程对命名管道具有 `FILE_CREATE_PIPE_INSTANCE` 访问权限。

同样，当客户端调用 `CreateFile()` 或 `CallNamedPipe()` 函数来连接命名管道的客户端端点时，系统会在授予客户端访问权限之前执行访问检查。

由 `CreateNamedPipe()` 函数返回的句柄始终具有 `SYNCHRONIZE` 访问权限。它还具有 `GENERIC_READ`、`GENERIC_WRITE` 或两者兼有，具体取决于管道的打开模式。以下是每种打开模式的访问权限：

| 打开模式                             | 访问权限                                                   |
| :----------------------------------- | :--------------------------------------------------------- |
| `PIPE_ACCESS_DUPLEX`（0x00000003）   | `FILE_GENERIC_READ`、`FILE_GENERIC_WRITE` 和 `SYNCHRONIZE` |
| `PIPE_ACCESS_INBOUND` （0x00000001） | `FILE_GENERIC_READ` 和 `SYNCHRONIZE`                       |
| `PIPE_ACCESS_OUTBOUND`（0x00000002） | `FILE_GENERIC_WRITE` 和 `SYNCHRONIZE`                      |

- `FILE_GENERIC_READ`：访问权限结合了从管道读取数据、读取管道属性、读取扩展属性以及读取管道的 DACL（自主访问控制列表）的权限。
- `FILE_GENERIC_WRITE`：访问权限结合了向管道写入数据、向其追加数据、写入管道属性、写入扩展属性以及读取管道的 DACL 的权限。

### 重叠模式（Overlapped Mode）

在重叠模式下，执行长时间读取、写入和连接操作的函数可以立即返回。这使得线程能够在后台执行耗时操作的同时执行其他操作。要指定重叠模式，请使用 `FILE_FLAG_OVERLAPPED` 标志。有关更多信息，请参阅 [同步和重叠输入和输出](https://learn.microsoft.com/zh-cn/windows/win32/ipc/synchronous-and-overlapped-input-and-output)。

`CreateFile()` 函数允许管道客户端使用 `dwFlagsAndAttributes` 参数为其管道句柄设置重叠模式（`FILE_FLAG_OVERLAPPED`）。

### 直写模式（Write-Through Mode）

使用 `FILE_FLAG_WRITE_THROUGH` 指定直写模式。 此模式仅影响不同计算机上的管道客户端和管道服务器之间的字节类型管道的写入操作。 在直写模式下，向命名管道写入数据的函数在通过网络传输数据到远程计算机的管道缓冲区之前不会返回。对于需要每次写入进行同步的应用程序非常有用。

如果未启用直写模式，系统会通过缓冲数据来提高网络操作的效率，直到累积了最小字节数或最长时间间隔为止。 缓冲使系统能够将多个写入操作合并到单个网络传输中。 这意味着，在系统将数据放入出站缓冲区之后，在系统通过网络传输数据之前成功完成写入操作。

`CreateFile()` 函数允许管道客户端使用 *`dwFlagsAndAttributes`* 参数为其管道句柄设置直写模式（`FILE_FLAG_WRITE_THROUGH`）。 创建管道句柄后，直写模式便无法更改。 对于同一管道实例，服务器句柄和客户端句柄的直写模式可以不同。

管道客户端可以使用 `SetNamedPipeHandleState()` 函数来控制禁用直写模式的管道传输前的字节数和超时期限。 对于只读管道，必须使用 `GENERIC_READ` 和 `FILE_WRITE_ATTRIBUTES` 访问权限打开管道句柄。

## 命名管道类型、读取和等待模式

管道服务器在 `CreateNamedPipe()` 函数的 `dwPipeMode` 参数中指定管道类型模式、读取模式和等待模式。管道客户端可以使用 `CreateFile()` 函数为其管道句柄指定这些管道模式。

### 类型模式

管道的类型模式决定了数据如何写入命名管道。数据可以通过命名管道以字节流或消息流的形式进行传输。管道服务器在调用 `CreateNamedPipe()` 创建命名管道的实例时指定管道类型。管道的所有实例必须具有相同的类型模式。

若要创建字节型管道，请指定 `PIPE_TYPE_BYTE` 或使用默认值。数据以字节流的形式写入管道，系统不会区分在不同写入操作中写入的字节。

若要创建消息类型管道，请指定 `PIPE_TYPE_MESSAGE`。 系统将每次写入管道的操作中所写入的字节视为一个消息单元。对于消息类型管道，系统始终将写入操作视为已启用直写模式。

### 读取模式

管道的读取模式确定如何从命名管道读取数据。 管道服务器在调用 `CreateNamedPipe()` 时指定管道句柄的初始读取模式。数据可以以字节读取模式或消息读取模式读取。字节类型管道的句柄只能处于字节读取模式。 消息类型管道的句柄可以处于字节读取模式或消息读取模式。 对于消息类型管道，同一管道实例的服务器句柄和客户端句柄的读取模式可以不同。

若要在字节读取模式下创建管道句柄，请指定 `PIPE_READMODE_BYTE` 或使用默认值。 数据作为字节流从管道中读取。读取操作在读取管道中的所有可用字节或读取指定字节数时成功完成。

若要在消息读取模式下创建管道句柄，请指定 `PIPE_READMODE_MESSAGE`。 数据作为消息流从管道中读取。 仅当读取整个消息时，读取操作才会成功完成。 如果指定的读取字节数小于下一条消息的大小，则函数在返回零之前读取尽可能多的消息（`GetLastError()` 函数返回 `ERROR_MORE_DATA`）。 可以使用另一个读取操作读取消息的剩余部分。

对于管道客户端，由 `CreateFile()` 函数返回的管道句柄最初始终处于字节读取模式。管道客户端和管道服务器都可以使用 `SetNamedPipeHandleState()` 函数更改管道句柄的读取模式。该管道句柄必须具有 `FILE_WRITE_ATTRIBUTES` 访问权限。

### 等待模式

管道句柄的等待模式决定了 `ReadFile()`、`WriteFile()` 和 `ConnectNamedPipe()` 函数如何处理长时间的操作。在阻塞等待模式下，这些函数会无限期地等待管道另一端的进程完成操作。在非阻塞等待模式下，这些函数会在原本需要无限期等待的情况下立即返回。

当管道为空时，`ReadFile()` 操作会受管道句柄等待模式的影响。 使用阻塞等待句柄，在将数据从写入管道的另一端的线程中获取之前，操作不会成功完成。 使用非阻塞等待句柄，函数将立即返回零，并且 `GetLastError()` 函数会返回 `ERROR_NO_DATA`。

当管道缓冲区中空间不足时，`WriteFile()` 操作会受管道句柄等待模式的影响。 使用阻塞等待句柄，写入操作必须等到另一端的线程读取数据从而在缓冲区中腾出足够空间后才能成功。 使用非阻止等待句柄时，写入操作会立即返回非零值，不会写入任何字节（对于消息类型管道）或在缓冲区空间允许的情况下写入尽可能多的字节（对于字节类型管道）。

当没有客户端连接或等待连接到管道实例时，`ConnectNamedPipe()` 操作会受到管道句柄等待模式的影响。对于阻塞等待句柄，连接操作不会成功，直到有管道客户端通过调用 `CreateFile()` 或 `CallNamedPipe()` 函数连接到管道实例。对于非阻塞等待句柄，连接操作会立即返回零，并且 `GetLastError()` 函数会返回 `ERROR_PIPE_LISTENING`。

默认情况下，由 `CreateNamedPipe()` 或 `CreateFile()` 函数返回的所有命名管道句柄均以阻塞等待模式创建。若要以非阻塞等待模式创建管道，管道服务器在调用 `CreateNamedPipe()` 时指定 `PIPE_NOWAIT`。

无论是管道客户端还是管道服务器，都可以通过在调用 `SetNamedPipeHandleState()` 函数时指定 `PIPE_WAIT` 或 `PIPE_NOWAIT` 来更改管道句柄的等待模式。

## 命名管道实例

最简单的管道服务器创建一个管道实例，连接到一个客户端，与该客户端通信，断开与客户端的连接，关闭管道句柄，然后终止。然而，管道服务器与多个管道客户端进行通信的情况更为常见。管道服务器可以通过依次连接和断开与每个客户端的连接来使用单个管道实例与多个管道客户端建立连接，但性能会很差。为了高效地同时处理多个客户端，管道服务器必须创建多个管道实例。

为多个管道实例提供服务有三种基本策略：

- 为每个管道实例创建单独的线程。 有关多线程管道服务器的示例，请参阅 [多线程管道服务器](https://learn.microsoft.com/zh-cn/windows/win32/ipc/multithreaded-pipe-server)。
- 通过在 `ReadFile()`、`WriteFile()` 和 `ConnectNamedPipe()` 函数中指定 `OVERLAPPED` 结构来使用重叠操作。 有关示例，请参阅 [使用重叠 I/O 的命名管道服务器](https://learn.microsoft.com/zh-cn/windows/win32/ipc/named-pipe-server-using-overlapped-i-o)。
- 通过使用 `ReadFileEx()` 和 `WriteFileEx()` 函数来执行重叠操作，这些函数指定了在操作完成时要执行的完成例程。 有关示例，请参阅 [使用完成例程的命名管道服务器](https://learn.microsoft.com/zh-cn/windows/win32/ipc/named-pipe-server-using-completion-routines)。

多线程管道服务器最容易编写，因为每个实例的线程仅处理单个管道客户端的通信。但每个线程都会使用系统资源，这对于处理大量客户端的管道服务器来说是一个缺点。

使用单线程服务器，可协调影响多个客户端的操作会更加容易，并且也更容易保护共享资源，避免被多个客户端同时访问。单线程服务器的挑战是，它需要协调重叠的操作来分配处理器时间以应对客户端同时产生的需求。
