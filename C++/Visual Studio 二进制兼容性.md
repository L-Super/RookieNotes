# Visual Studio 二进制兼容性

Visual Studio 2013 及更早版本中的 Microsoft C++ （MSVC） 构建工具不能保证主要版本的二进制兼容性。 链接由这些构建工具的不同版本生成的目标文件、静态库、动态库和可执行文件。应用程序二进制接口（ABI）、对象格式和运行时库都是不兼容的。

 Visual Studio 2015 及更高版本中改变了此行为。 由其中**任一版本的编译器编译的运行时库和应用具有二进制兼容性。** 它反映在 MSVC 构建工具的主版本号中，该编号自 Visual Studio 2015 起的所有版本都从 14 开始。 （Visual Studio 2015 的构建工具版本为 v140，2017 为 v141，2019 为 v142，2022 为 v143）。 假设你有由 Visual Studio 2015 构建的第三方库。 仍可在 Visual Studio 2017、2019、2022 或 2026 生成的应用程序中使用它们。 无需使用匹配版本重新编译。 最新版本的 Microsoft Visual C++ 可再发行程序包（C++ Redistributable package）适用于所有版本。

## 对二进制兼容性的限制

v140、v141、v142、v143 和 v145 构建工具与次要版本更新之间存在三个重要的二进制兼容性限制：

- 可以组合使用不同版本的 v140、v141、v142、v143 和 v145 构建工具创建的二进制文件。 关键规则是链接器只能处理由**与自身版本相同（或更早版本）**的构建工具所生成的输入文件。这适用于应用、导入库、静态库和 LINK 输入文件中所述的其他文件。 在某些情况下，使用较新版本构建工具为隐式链接的 DLL 生成的导入库，可以由较早版本的构建工具进行链接——尤其当该导入库在导入/导出时严格使用 `extern "C"` 时。 以下是这些规则的示例：
  - 使用 2017 构建工具（v141，版本 15.0 到 15.9）编译的应用程序，可以链接使用 Visual Studio 2022 版本 17.8（v143）编译的静态库，但链接必须使用 17.8 或更高版本的构建工具完成。
  - 使用 VS 2015、2017、2019、2022 和 2026 构建的应用程序和库可以相互链接，但链接操作所使用的构建工具版本，必须不低于建这些二进制文件时使用的最新构建工具版本。例如，有三个二进制文件分别使用 VS 2015 版本 14.3、VS 2017 版本 15.9 和 VS 2019 版本 16.11 的构建工具生成，可以使用任意 16.11 或更高版本的构建工具将它们链接在一起。
  - 如果某个 DLL 是使用较新版本的构建工具构建的，且其所有导出均遵循 C 语言调用约定（`extern "C"`），则其导入库有时可与较旧版本的构建工具配合使用。但官方唯一明确支持的场景，是在使用较旧构建工具的情况下使用较新版本的 Windows SDK。
- 应用使用的可再发行组件具有类似的二进制兼容性限制。 混合由不同支持版本的生成工具生成的二进制文件时，可再发行组件版本必须至少与任何应用组件使用的最新生成工具一样新。
- 使用 [`/GL`（完整程序优化）](https://learn.microsoft.com/zh-cn/cpp/build/reference/gl-whole-program-optimization?view=msvc-170)编译器选项编译或使用 [`/LTCG`（链接时代码生成）](https://learn.microsoft.com/zh-cn/cpp/build/reference/ltcg-link-time-code-generation?view=msvc-170)链接的静态库或对象文件，在各个版本之间（包括次要版本更新）没有二进制兼容性。 使用 **`/GL`** 和 **`/LTCG`** 编译的所有目标文件和库必须使用完全相同的构建工具进行编译和最终链接。 例如，使用 **`/GL`** Visual Studio 2019 版本 16.7 生成工具生成的代码无法链接到在 **`/GL`** Visual Studio 2019 版本 16.8 生成工具中生成的代码。 编译器会发出[严重错误 C1047](https://learn.microsoft.com/zh-cn/cpp/error-messages/compiler-errors-1/fatal-error-c1047?view=msvc-170)。
